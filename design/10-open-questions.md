# Open Questions — All Resolved

All OQ-1 through OQ-11 are locked decisions.

## OQ-1: Deployment — Docker Compose ✅

Docker Compose for both local and production. K8s is not appropriate at current scale — it solves problems (horizontal scaling, multi-cluster orchestration, rolling deployments) the NUC doesn't have. `restart: always` handles container recovery. Migration path when ready: Docker Compose → Docker Swarm (multi-machine) → K8s (cloud-scale). `kompose convert` turns the Compose file into K8s manifests when needed.

Local dev: native Python venv + local Postgres/Redis is also fully supported. `uvicorn --reload` for hot-reload during backend development. A `docker-compose.dev.yml` variant runs the full stack for testing worker behavior.

## OQ-2: Database migrations — Alembic, autogenerate with mandatory review ✅

Alembic tracks schema changes as versioned migration files applied in order.

```bash
# Change SQLAlchemy model → generate migration
alembic revision --autogenerate -m "add entry_end_date to entries"
# Review generated file before running (autogenerate misses renames, some indexes)
alembic upgrade head   # applies to local DB; same command on prod
```

**Never apply autogenerated migrations without reading the generated SQL** — renames appear as drop+add (data loss risk), and column type diffs occasionally produce noise.

## OQ-3: Local dev environment ✅

Both options supported and equivalent. Native venv is faster for day-to-day feature development; Docker Compose locally is better for integration testing and pre-deploy verification.

## OQ-4: Shared code between Next.js and Expo — separate codebases ✅

SSR requirement for OG previews makes a single Expo Web codebase impossible. Separate apps. A shared `packages/api-types` package (TypeScript types auto-generated from FastAPI's OpenAPI schema via `openapi-typescript`) keeps the API contract in sync.

---

## OQ-5: TypeScript ✅

TypeScript for both Next.js and Expo. FastAPI generates OpenAPI schema; `openapi-typescript` generates TS types for frontends. Catches API contract mismatches at compile time.

---

## OQ-6: Next.js auth — FastAPI as source of truth ✅

FastAPI handles all OAuth flows and JWT issuance. Next.js SSR forwards the user's access token in API requests. No NextAuth. Auth is unified across web and mobile.

## OQ-7: Diary URL slugs ✅

Add `diaries.slug text UNIQUE NOT NULL` at schema creation. Slugs derived from diary name at creation, disambiguated on collision: `eleanors-diary`, `eleanors-diary-2`. Display URLs: `diary.perfectday.bdsys.net/d/{slug}/entries/{entry_date}`. Internal IDs remain UUIDs.

**Change early** — changing URLs later breaks shared links.

## OQ-8: Monorepo ✅

Single repo with pnpm workspaces:

```
perfect-day/
  apps/
    api/          (FastAPI — Python)
    web/          (Next.js — TypeScript)
    mobile/       (Expo — TypeScript, Phase 2)
  packages/
    api-types/    (openapi-typescript generated; shared by web + mobile)
  design/         (these planning docs)
  docker-compose.yml
  docker-compose.dev.yml
```

Add Turborepo if build times become painful.
